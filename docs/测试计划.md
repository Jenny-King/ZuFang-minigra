# 测试计划

## 1. 文档目标

- 为当前租房小程序提供一份可执行、可回归、可交接的测试计划
- 将测试活动按“测试层次 + 业务链路”组织，避免只测单点功能
- 统一开发、自测、联调、答辩演示时的测试入口、数据准备和验收口径

## 2. 测试范围

本计划覆盖以下模块：

- 认证与账号：微信登录、手机号注册/登录、重置密码、绑定/解绑微信、换绑手机号、绑定邮箱、注销账号
- 找房链路：首页列表、区域/租金/户型筛选、搜索、详情页、地图能力
- 房源链路：房东发布、编辑、上下架、删除、我的房源
- 用户行为链路：收藏、浏览历史、通知、个人中心、角色切换
- 沟通链路：发起会话、发送消息、通知已读、客服反馈
- 后端支持链路：云函数、会话鉴权、测试数据初始化与清理

本计划暂不重点覆盖：

- 线上生产环境压测
- 大规模并发性能专项
- 真机兼容性矩阵扩展到多个品牌型号

## 3. 测试目标

### 3.1 业务目标

- 核心租房链路能够闭环跑通：注册/登录 -> 找房 -> 详情 -> 收藏/历史 -> 联系房东 -> 通知/客服
- 房东链路能够闭环跑通：登录 -> 发布 -> 编辑 -> 上下架/删除 -> 处理聊天
- 设置链路不会破坏当前会话与身份映射：修改密码、换绑手机号、绑定微信、注销账号后状态一致

### 3.2 技术目标

- 页面层、服务层、云函数、数据库之间的调用边界符合架构设计
- 所有受保护接口均通过 `accessToken -> user_sessions -> userId` 鉴权
- 自动化测试、手动回归、数据脚本之间职责清晰，不互相污染

## 4. 测试环境与入口

### 4.1 环境要求

- 使用当前开发云环境
- `bootstrap` 相关初始化、清理能力仅允许在显式非生产环境执行
- 执行 UI 自动化前，必须确认 `config/env.js` 未指向生产环境

## 5. 测试角色与数据

建议至少准备以下三类账号：

- 房东账号：用于发布、编辑、上下架、删除、接收聊天
- 租客账号：用于找房、收藏、历史、发起聊天、查看通知
- 备用租客账号：用于双端聊天、绑定/解绑微信、账号切换场景

数据准备要求：

- 房源数据至少保留 6 条，覆盖不同区域、租金段、户型
- 聊天测试前，至少保证 1 个租客和 1 个房东账号可正常登录
- 通知测试前，至少完成 1 次反馈提交或聊天消息流转
- 如数据已污染，优先执行测试账号清理与房源数据重建

## 6. 测试分层策略

### 6.1 单元测试

目标：

- 验证纯函数、辅助模块、格式转换和校验逻辑

当前重点：

- 聊天消息解析与会话预览文案
- 设置页手机号校验与交互文案

准入要求：

- 新增纯工具逻辑必须优先补单元测试
- 影响输入校验、状态映射、消息解析的变更必须回归

当前状态：

- 仓库中已不再保留 `test-scripts/unit/` 单元测试脚本
- 如需恢复该层测试，请在新的测试目录重新建立 Jest 用例

### 6.2 云函数测试

目标：

- 验证 action 入参、权限、状态流转、错误码、写库行为

当前重点：

- `auth`
- `user`
- `house`
- `chat`
- `favorite`
- `history`
- `support`
- `bootstrap`
- `map`

准入要求：

- 新增 action 或修改鉴权逻辑时，必须补充对应 `__tests__`
- 会影响 `user_sessions`、`user_identities`、`users` 的改动必须优先回归

### 6.3 UI 自动化测试

目标：

- 验证页面渲染、主要交互、跳转、空态、异常态

当前状态：

- `test-scripts/` 下仅保留历史脚本 `legacy/ui-home-2.js`
- 原自动化场景脚本与底座已移除，不再作为发布前准入项

准入要求：

- 影响主页面交互、表单流程、关键跳转时，应补充新的 UI 回归方案

### 6.4 手动回归测试

目标：

- 覆盖双账号协作、微信身份绑定、跨页面状态同步、云数据真实写入等自动化不易稳定覆盖的场景

重点：

- 微信绑定/解绑
- 双账号聊天
- 角色切换后的页面权限变化
- 注销账号后的多端状态一致性

## 7. 主链路测试设计

以下链路按真实使用顺序组织，测试时优先保证“先有入口，再有中间状态，最后有结果落点”。

### 7.1 链路 A：注册/登录到进入首页

目标：

- 验证新用户可以完成注册或登录，并获得稳定会话

覆盖步骤：

1. 打开登录或注册页
2. 输入手机号并完成验证码/密码流程
3. 注册成功后自动登录，或登录成功后进入首页
4. 校验本地 `accessToken + userInfo` 已建立
5. 刷新当前用户信息，验证会话未丢失

关键断言：

- 成功后首页正常打开
- `user_sessions` 生效
- `wechatLogin`、`register`、`loginWithPassword`、`loginWithPhoneCode` 返回结构一致

### 7.2 链路 B：找房、筛选、查看详情

目标：

- 验证租客可以从首页完成一次有效找房

覆盖步骤：

1. 打开首页并等待首屏数据加载
2. 执行城市/区域/租金/户型筛选
3. 校验列表刷新、空态与重置筛选
4. 点击房源卡片进入详情页
5. 校验详情信息、图片、房东信息、联系入口

关键断言：

- 列表与筛选条件同步
- 空态与恢复路径清晰
- 详情页打开后不报错，关键字段完整

### 7.3 链路 C：详情触发收藏与浏览历史

目标：

- 验证用户行为数据可被正确记录和回显

覆盖步骤：

1. 从首页进入房源详情
2. 点击收藏
3. 返回个人中心进入收藏列表
4. 再次进入详情或浏览其他房源
5. 打开浏览历史列表验证记录

关键断言：

- 收藏状态即时刷新
- 收藏列表可见对应房源
- 浏览历史产生且排序正确
- 删除或清理后状态同步更新

### 7.4 链路 D：从详情发起聊天到通知

目标：

- 验证租客与房东之间的沟通链路闭环

覆盖步骤：

1. 租客从详情页发起聊天
2. 系统创建或复用会话
3. 租客发送消息
4. 房东侧进入会话列表与详情页查看消息
5. 验证通知、未读状态、已读状态更新

关键断言：

- `createConversation` 能返回有效会话
- `sendMessage` 成功写入消息
- 会话列表预览文本和未读数正确
- 通知页只展示通知，不混入聊天消息

### 7.5 链路 E：客服反馈到系统通知

目标：

- 验证用户反馈提交后能收到系统回执

覆盖步骤：

1. 用户进入客服中心
2. 提交问题反馈
3. 校验反馈提交成功
4. 进入通知页查看系统回执

关键断言：

- `support_feedbacks` 成功写入
- `messages(type=system)` 同步生成
- 通知列表正确展示回执文案

### 7.6 链路 F：房东发布、编辑、上下架、删除

目标：

- 验证房东业务链路完整可用

覆盖步骤：

1. 房东登录并进入发布/管理页
2. 新建房源并完成三步发布
3. 返回管理页确认房源出现
4. 编辑房源并校验回填
5. 执行上下架或删除
6. 返回首页确认展示结果符合预期

关键断言：

- 创建后 `landlordUserId` 正确
- 编辑不丢字段
- 删除/上下架后列表及时刷新
- 首页与个人房源管理页状态一致

说明：

- `package-profile/pages/my-houses` 当前未在 `app.json` 注册，因此“我的房源”链路以现有发布管理入口为准进行验证

### 7.7 链路 G：设置与账号安全

目标：

- 验证设置类操作不会破坏身份映射和登录态一致性

覆盖步骤：

1. 进入设置页
2. 修改密码并重新登录验证
3. 换绑手机号并验证新手机号登录
4. 绑定/解绑微信并检查状态文案
5. 绑定邮箱
6. 注销账号并验证旧会话失效

关键断言：

- 修改密码后旧密码失效、新密码生效
- 换绑手机号不会生成脏用户
- 微信身份不会重复绑定到多个账号
- 注销后 `users`、`user_identities`、`user_sessions` 状态一致

## 8. 非功能与风险测试

### 8.1 权限与安全

- 未登录访问受保护接口应返回 `401`
- 不同角色执行越权写操作应返回 `403`
- 已过期或已撤销会话不得继续访问业务接口

### 8.2 数据一致性

- 删除房源后收藏、历史、会话关联展示需要验证是否可正常降级
- 收藏、历史、通知数量应与个人中心头部统计保持一致
- 聊天消息预览文案与详情页消息内容必须一致

### 8.3 兼容与稳定性

- 网络失败时，表单提交和聊天发送要有失败反馈
- 空列表、空态、慢加载、重复点击需要有稳定表现
- 重新登录、角色切换、切后台再回来后状态不能错乱

## 9. 执行顺序建议

### 9.1 日常开发自测

1. 跑受影响模块的单元测试
2. 跑对应云函数测试
3. 跑受影响页面的 UI 脚本
4. 手动验证一条完整主链路

### 9.2 提测前回归

1. 重新部署受影响云函数
2. 清理脏测试账号与脏房源数据
3. 重建基础房源数据
4. 执行 `npm test -- --runInBand`
5. 执行 `node test-scripts/core/run-all.js`
6. 按链路 A -> G 完成手动回归

### 9.3 答辩演示

1. 先演示租客登录与找房
2. 演示详情、收藏、历史、聊天
3. 演示通知与客服反馈
4. 切换到房东账号演示发布与管理
5. 最后演示设置能力与账号安全能力

## 10. 准入与准出标准

### 10.1 提测准入

- 受影响功能已完成自测
- 对应单元测试和云函数测试通过
- 至少一条主链路跑通
- 无阻塞级别已知问题

### 10.2 发布准出

- 核心主链路 A、B、D、F、G 全部通过
- 反馈链路 E 通过
- 自动化测试与手动回归无 P0/P1 问题
- 测试数据、测试账号、环境标识可追溯

## 11. 缺陷分级建议

| 级别 | 定义 | 示例 |
| --- | --- | --- |
| `P0` | 核心链路阻断，无法演示或无法使用 | 无法登录、无法发布、无法进入详情 |
| `P1` | 主流程可进入但关键结果错误 | 收藏不生效、聊天发不出去、通知错乱 |
| `P2` | 非核心功能异常或存在明显体验问题 | 空态错误、字段回填不完整、倒计时异常 |
| `P3` | 文案、样式、边角一致性问题 | 提示文案不统一、样式错位 |

## 12. 维护要求

- 新增业务能力时，必须在本计划中补充所属链路和测试入口
- 新增自动化脚本时，需同步更新 `docs/测试与联调指南.md`
- 若架构、鉴权或目录结构变更，需同步更新本计划和 `docs/技术架构设计.md`
