# 历史疑惑与当前结论

> 本文件保留项目推进过程中的关键问题和最终结论，供后续排查时快速回看。

## 1. 为什么曾经出现“新账号和旧账号 `_openid` 一样”？

原因不是手机号账号复用，而是：

- 在同一个微信 / DevTools 运行身份下注册多个手机号账号
- 云函数拿到的 `OPENID` 实际上是“当前微信身份”的标识
- 旧实现把 `_openid` 既当作微信绑定标识，又当作“当前登录用户是谁”

结果：

- 多个 `users` 记录共享同一个 `_openid`
- 服务端一旦按 `_openid` 查当前用户，就会串号

## 2. 现在为什么要拆成 `users + user_identities + user_sessions`？

为了把三件事彻底分开：

- `users`：这个用户是谁，资料是什么
- `user_identities`：这个用户能用什么身份登录或绑定
- `user_sessions`：这个用户当前有哪些有效登录会话

对应关系：

- `accessToken` 对应一个 session
- session 对应一个 `userId`
- `userId` 再对应用户资料

## 3. 当前认证结论

- 手机号注册 / 登录不再自动绑定微信身份
- 微信绑定必须显式调用 `bindWechat`
- 微信解绑必须显式调用 `unbindWechat`
- 受保护接口统一走 `accessToken -> user_sessions -> userId -> users`
- 前端统一保存 `accessToken + userInfo`

## 4. 为什么 `updateProfile` 不再允许直接改手机号？

因为手机号已经不只是资料字段，也是登录身份的一部分。

如果在 `users.phone` 改了手机号，但没有同步更新 `user_identities(type=phone)`，就会再次出现数据不一致。

因此当前结论是：

- `updateProfile` 只允许改昵称、微信号、地区、头像等资料
- 手机号换绑必须单独设计验证码校验和身份迁移流程

## 5. 如果要做双账号真实联调，需要注意什么？

- 两个业务账号可以用不同手机号
- 但如果要测试“微信绑定”或依赖微信身份的能力，必须使用不同微信运行身份
- 不能把同一个微信身份同时绑定给两个业务账号

## 6. 当前已落地的处理

- 新增 `user_identities`
- 新增 `user_sessions`
- 登录成功返回 `accessToken + userInfo`
- 页面刷新当前用户时统一走 store，token 失效会自动清理本地状态
- 云端初始化集合时已加入 `user_identities` 和 `user_sessions`
