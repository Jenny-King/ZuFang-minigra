# 租房微信小程序开发规范说明

## 1. 总原则

- 保持目录清晰
- 保持职责单一
- 保持命名统一
- 保持调用链简单
- 保持页面层轻量
- 保持认证链路统一

## 2. 技术规范

- 使用微信小程序原生开发
- 不使用 `uni-app`、`Taro` 等跨端框架
- 使用微信云开发作为后端基础设施
- 所有异步逻辑统一使用 `async/await`
- 页面禁止直接调用 `wx.cloud.callFunction`
- 页面禁止直接访问数据库

## 3. 分层规范

### 3.1 页面层

- 只负责页面状态、事件响应、渲染
- 不写复杂业务处理
- 不直接依赖后端底层数据结构
- 不自行实现鉴权逻辑

### 3.2 服务层

- 所有业务请求必须放在 `services` 中
- 一个 service 只负责一个业务领域
- 统一处理参数、错误和返回值
- 统一通过 `services/cloud/call.js` 注入 `auth.accessToken`

### 3.3 状态层

- 登录态统一管理 `accessToken + userInfo`
- 页面刷新用户资料时优先通过 `userStore.refreshCurrentUser()`
- token 失效后必须清理本地会话

### 3.4 工具层

- 只放纯函数和通用封装
- 不要将复杂业务逻辑放入 `utils`

## 4. 命名规范

### 4.1 文件命名

- 页面目录使用明确英文语义
- service 文件统一使用 `*.service.js`
- store 文件统一使用 `*.store.js`

### 4.2 变量命名

- 使用小驼峰命名
- 布尔变量以 `is`、`has`、`can` 开头
- 数组变量使用复数名词

### 4.3 常量命名

- 全大写加下划线
- 放在 `config/constants.js`

## 5. 代码风格

- 优先早返回，减少层层嵌套
- 避免超长函数
- 单个函数尽量只做一件事
- 重复逻辑提取到服务层或工具层
- 错误处理必须明确，不能静默吞错

## 6. 页面开发规范

- 页面不得直接使用云函数名
- 页面不得直接写数据库集合名
- 页面请求逻辑必须抽到 service
- 页面加载、空态、错误态要明确
- 受保护页面必须通过 `authUtils.requireLogin()` 或 store 鉴权

## 7. 云函数开发规范

- 云函数按领域拆分
- 每个云函数内部使用 `action` 分发
- 返回结果统一结构
- 受保护接口必须先解析 `auth.accessToken`
- 当前用户识别必须通过 `user_sessions -> userId -> users`
- 必须校验参数合法性
- 敏感字段不直接返回前端

## 8. 身份与认证规范

- `users` 只存用户资料和认证状态
- `user_identities` 只存手机号 / 微信身份映射
- `user_sessions` 只存登录会话
- `userId` 是所有业务关系的唯一用户主键
- 禁止重新引入 `_openid` 作为“当前用户识别”依据
- 手机号注册 / 登录不得自动绑定微信身份
- 微信绑定必须通过显式 `bindWechat`
- 微信解绑必须通过显式 `unbindWechat`

## 9. 数据规范

- 业务关联统一使用 `userId`
- 时间统一使用 `Date`
- 密码统一使用哈希存储
- 身份证信息统一加密存储、脱敏输出
- `updateProfile` 不允许直接修改手机号
- 不允许将测试数据逻辑暴露到正式业务页面

## 10. 验证规范

- 每改完一个模块都要可预览
- 每改完一个页面都要检查控制台报错
- 每改完一个云函数都要验证返回值结构
- 认证链路改动后至少回归：
  - 微信登录
  - 手机号注册 / 登录
  - `getCurrentUser`
  - 绑定 / 解绑微信
  - 退出登录
- 删除代码前先确认无实际引用

## 11. 禁止事项

- 禁止页面直连云函数
- 禁止在 `app.js` 中自动插入测试数据
- 禁止让 `users` 同时承担身份映射和会话识别职责
- 禁止通过 `updateProfile` 偷改手机号
- 禁止在一个文件里堆积过多跨业务逻辑
